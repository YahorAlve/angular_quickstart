1. Java classes structure

Lets take a look at the next picture. Here is example of structure of java classes which is very convenient from our point of view for creating and supporting UI tests. And there are no obstacles to implement your structure on your own. You will be also provided with information about how to map Jbehave steps onto the java code in the Mapping Steps onto the Java code.

Further we are going to run through all these folders.

1.1. Runner class

Runner class are necessary for jbehave-maven-plugin as an entry point. See please the parameter <jp.stories> in [Jbehave maven plugin]. The plugin launches your tests and it needs configuration class. We created it for you and all you need is just extended our StoriesConfiguration.class as it is showed on the picture. You can even overload some methods and it could change logic for running tests. But it demands the strong knowledge of Jbehave library. We recommend you not perform this unless you realize what you do.

1.2. Mapping Steps onto the Java code

It is very important part. Here is shown how to map your text steps onto java methods. Lets take a look at the next picture.

As We said before here [Spring Framework], We are using here the Spring Dependency injection implementation. I try to lay out it simply without deep diving in this. It is like a container with a lot of java objects are created from your classes. These objects are SINGLETONS.

Note

We hope you know what singleton is and which tricks could appear during using them. For example using singletons in multythreading - It is one object and if you change something in it other thread will get it with your changes.

In order to put your object into there you have to write the next annotation @Component :

...
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {
....
}

If you would like to pull it or other objects out and start using them in your code please write another annotation @Autowired :

...
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }
    ....
}

Why do you need to put your Java Steps Class into container ? Because of Jbehave internal mechanism of mapping text steps onto java methods. Simply put Jbehave parse your step for example "When open google page" :

...
Scenario: google search page
When open google page
...

Then it goes to the "Container" and run through all methods with @When @Then annotations of the located there singletons trying to find necessary methods. Here it is @When("open google page").

...
import org.jbehave.core.annotations.Then;
import org.jbehave.core.annotations.When;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }

    @When("open google page")
    public void openGooglePage() {
        getPage().open();
        getPage().waitPageLoaded();
    }

    ....
}
Note

Therefore If inside container there is 2 or more the same methods it takes the first found one. If there is no appropriate method, you will be provided with exceptions.

1.3. Java page classes or Page Object pattern

1.3.1. Page Objects

As We said before here [Selenium WebDriver], we are using WebDriver for our UI testing. There is a very famous and convenient pattern for working in close conjunction with it. It is called Page Object. It is very easy to figure it out. Your class just represents some Web Page, in other words Web Page is described by java code. Just take a look at the next picture and it gets a lot clearer.

As you see from the picture all you need to turn the Java class into the Page object is to extend our AbstractPage.class :

...
import com.epam.e3s.test.ui.framework.pages.AbstractPage;
import org.jbehave.web.selenium.WebDriverProvider;

public class GoogleFirstPage extends AbstractPage{

    public GoogleFirstPage(WebDriverProvider driverProvider) {
        super(driverProvider);
    }
    ...
}
...

Then you can get this page in any place of code by writing the next :

...
import com.epam.e3s.test.ui.framework.pages.Pages;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }
    ....
}

Class Pages is a container for stocking all pages. When you creates your page first time it process your class, create Page Object, and put result into container. In further you will receive this page from container without extra workaround. By extending of AbstractPage.class you also get a lot useful methods for your work.

If you pay your attention to the names of methods, you are very likely to understand what they perform. Below We explore a few from them :

  • Open Page

...
public class GoogleFirstPage extends AbstractPage {
    ...

    public GoogleFirstPage open() {
        String urlBase = propertyHolder.getValue(PropertyConstants.GOOGLE_FIRST_PAGE);
        get(urlBase);
        return this;
    }
    ...
}

As you understand this method open page in browser. It performs the next method is taken from ancestor :

...
        get(urlBase);
...

  • Wait Page Load

...
    public static final String SEARCH_BAR_XPATH = "//input[@id='lst-ib']";

    public void waitPageLoaded() {
            waitVisibilityOfElementLocated(By.xpath(SEARCH_BAR_XPATH),
                                           PropertyConstants.COMMON_WAIT);

...

Often it is very convenient to wait for Page will be loaded completely and only after it happens to try performing some actions on this page, such as getting some elements, pushing buttons and so on. And you as a developer of tests are much better aware what exactly "Page has been loaded" means for specific page. For example some bottom page elements appear or all javascript code has been performed and so on. In example we wait emergency of search bar on the page.

Here you also can see another method is taken from ancestor :

...
    waitVisibilityOfElementLocated(By.xpath(SEARCH_BAR_XPATH), PropertyConstants.COMMON_WAIT);
...

It is quite clear that this method waits some time until element gets visible on the page. If element doesn’t become visible during this time, you will be provided with an exception. Lets move on to the next methods. You can see the emergency of new interesting returned by methods object - WebElement.

1.3.2. Web Elements

What is exactly Web page? Basically It is suite of high number of Web Elements. Such as buttons, inputs, dialog windows, paginators, bookmarks and so on, we can continue endlessly. Selenium WebDriver provides another abstraction which is called WebElement.class. It has some basic methods for working with it. You can browse these methods on your own. We just provide a little example of interaction with this element. So lets go on with code is taken from Mapping Steps onto the Java code. Put your attention to 2 last methods:

...
@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }

   @Then("write text '$text' into search bar")
    public void writeQuery(String text) {
        getPage().getSearchBar().sendKeys(text);
    }

    @Then("click search button")
    public void clickSearchButton() {
        getPage().getSearchButton().click();
    }
}

As you see we are getting WebElements from page and try to interact with them. Everything is very easy. In other words you say to page : "The page give me the webElement search bar" and then you interact with it by calling sendKeys() - typing text inside search bar. Then you say to page : "Now The page give me the webElement search button" and push this button by calling click().

Now lets have a look at these methods which return our WebElements :

...
import org.openqa.selenium.By;
import org.openqa.selenium.WebElement;

/**
 * Created by Yahor_Alve on 4/1/2016.
 */
public class GoogleFirstPage extends AbstractPage{

    public static final String SEARCH_BAR_XPATH = "//input[@id='lst-ib']";
    public static final String SEARCH_BUTTON_XPATH = "//input[@name='btnK']";

    public GoogleFirstPage(WebDriverProvider driverProvider) {
        super(driverProvider);
    }

    public WebElement getSearchBar() {
        return getVisibleElementLocated(By.xpath(SEARCH_BAR_XPATH),
                                        PropertyConstants.COMMON_WAIT);
    }

    public WebElement getSearchButton() {
        return getVisibleElementLocated(By.xpath(SEARCH_BUTTON_XPATH),
                                        PropertyConstants.COMMON_WAIT);
    }

}

In getSearchBar() we return WebElement Search Bar. Actually we do not know what is the element, it is just abstraction. We just wait for getting visible this element during some time and return this object without any clue what it can do or not - It supposes that who asked for this element is aware what this element do. In example above you saw that we typed text inside this element.

Also you can see such phrase as By.xpath(SEARCH_BAR_XPATH). What is Xpath? Please see very good and understandable article : http://www.w3schools.com/xsl/xpath_nodes.asp. Simply put It is address a wanted WebElement on the page. Knowledge about structuring it correctly is very important - but if you are QA and test Web part of application You must be familiar with it.

Note

Please follow the next recommendation. You are not supposed to place some assertions or tests in the Page. For separating logic and making the principle of interaction different layers of your code understandable for others, we propose you bear the next statement in your mind. The page is able only to perform actions are relevant to web page - opening, waiting for loading, giving WebElements, occasionally performing some difficult operations related only with this page. NOT ANY assert checks! They are performed by Java steps layer Mapping Steps onto the Java code like below :

...
import org.testng.Assert;

@Component
public class GoogleSecondPageSteps {
    ...

    @Then("the title of '$number' result on the Google search page is '$result'")
    public void checkingTitleFirstResult(String number, String result) {
        Assert.assertEquals(getPage().getResultTitle(number), result);
    }
    ...
}
