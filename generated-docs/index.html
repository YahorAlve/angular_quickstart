The Framework was created to simplify the process of introducing and developing UI testing on any projects. It combine a few open source and well known in the programmers' world frameworks, provide you with simplified API and hired complex logic for unexperienced users.

Guts of the Framework

This framework include the next libraries:

  • Jbehave (http://jbehave.org/)

  • Selenium WebDriver (http://www.seleniumhq.org/projects/webdriver/)

  • Spring Framework (https://spring.io/docs)

  • BrowserMob Proxy (https://bmp.lightbody.net/)

  • Report Portal (https://kb.epam.com/display/EPMRPP/Report+Portal)

1. Jbehave

It is a heart of our framework. All other infrastructure was created around it. This engine launches tests, deals with the process of passing them and collects all relevant and necessary information. The best way to become familiar with it is to visit http://jbehave.org/. As soon as you come there you see main features of this framework. Simply put, it allows you to create text files in non-programing language and map it onto the java code. It is also based on BDD (Behavior-Driven Development) testing approach, but you are not supposed to follow it.

2. Selenium WebDriver

It is very famous framework for testing UI. You were very likely to hear about it, if you do not, there are no problems to get to know with it. There is a huge amount of information on the Internet are dedicated to it. Concisely, it let you write code in some programing language (Java, Python e.g.) in order to interact with your browsers (Firefox, Chrome e.g.). You do not need to know something more than language are used you. It takes care about whole process of interacting in the most similar way as if you do it from your keyboard and mouse whereas you just create simple strings of code in understandable for programmers’ way. One else important thing is it communicates with browser via REST by sending clear commands for browser. In other words it is converter from for example java code to machine code for browsers.

3. Spring Framework

There are a few ways in Jbehave how to implement mapping steps (text is written in English) onto the code. We selected from all those ways the implementation of IoC (Inversion of Control) via Dependency Injection, namely via Spring using the SpringStepsFactory. (See http://jbehave.org/reference/stable/dependency-injection.html Using Spring). We also use this feature(DI) for our framework in some places where it had sense. More over we used aspect annotations. In addition Spring was appointed to be responsible for handling properties in code.

4. BrowserMob Proxy

Some times you need to insert necessary arguments inside header. For example user name for sites with basic authorization. It was strictly forbidden for WebDriver to provide such possibility. But this open source project allow you to perform it. It just runs light weight proxy server which intercepts requests from your browser and transform them in needed for you way. See please Useful steps.

5. Report Portal

It is internal application of Epam systems. We use this very convenient tool for reporting our test results. We can track all our outcomes in constantly working CI process very well. All results of tests are pushed into https://rp.epam.com in real time and stocked there. All failed tests contains screenshot is done at the time emergency of fault. Everyone can come there and see the results.For more information please see https://kb.epam.com/display/EPMRPP/Report+Portal.

7. Pom.xml

Lets have a look at pom.xml. You could just copy its content and adjust it in more convenient way from your point of view. I won’t explain how maven works, you are supposed to be familiar with it. I would like to emphasize a few main pieces of this pom which are necessary for launching your tests.

7.1. Dependency

The framework maven dependency has to be included into your pom.xml.

<dependency>
    <groupId>com.epam.e3s</groupId>
    <artifactId>ui-test-framework</artifactId>
    <version>${ui-test-framework}</version>
</dependency>

For the reason that the framework is a proprietary product of EPAM systems The artifacts of it are located in the EPAM internal repository. In order to pull it you have to add external repo in the following way :

<repositories>
    <repository>
        <id>testing</id>
        <url>http://nexus.e3s.epam.com/content/repositories/testing/</url>
    </repository>
</repositories>

If You would like to use versions in developing stage you need to add SNAPSHOT repository :

<repositories>
    <repository>
        <id>snapshot</id>
        <url>http://nexus.e3s.epam.com/content/repositories/snapshots/</url>
    </repository>
</repositories>

7.2. Build part

7.2.1. Jbehave maven plugin

The main thing to note is the build part of this pom. It is responsible for starting you tests and giving criteria for the launch. Take a look at plugin jbehave-maven-plugin. As I mentioned before Jbehave engine handles the most part of work, so not surprisingly that there is a special plugin for these targets.

Plugin jbehave-maven-plugin. It hast to be inserted in your pom.xml.

<plugin>
    <groupId>org.jbehave</groupId>
    <artifactId>jbehave-maven-plugin</artifactId>
    <version>${jbehave.core.version}</version>
    <executions>
        <execution>
            <id>unpack-view-resources</id>
            <phase>process-resources</phase>
                <goals>
                    <goal>unpack-view-resources</goal>
                </goals>
        </execution>
        <execution>
            <id>embeddable-stories</id>
            <phase>integration-test</phase>
            <goals>
                <goal>run-stories-as-embeddables</goal>
            </goals>
                <configuration>
                    <includes>
                        <include>${jp.stories}</include>
                    </includes>
                    <excludes/>
                    <scope>compile</scope>
                    <outputDirectory>${project.build.outputDirectory}/jbehave</outputDirectory>
                    <generateViewAfterStories>
                        ${jp.generate.view.after.stories}
                    </generateViewAfterStories>
                    <ignoreFailureInStories>${jp.ignore.failures}</ignoreFailureInStories>
                    <ignoreFailureInView>${jp.ignore.failures}</ignoreFailureInView>
                    <storyTimeoutInSecs>${jp.story.timeout.in.secs}</storyTimeoutInSecs>
                    <threads>${jp.threads}</threads>
                    <skip>${jp.skip}</skip>
                </configuration>
        </execution>
    </executions>
    <dependencies>
        <dependency>
            <groupId>log4j</groupId>
            <artifactId>log4j</artifactId>
            <version>${log4j.version}</version>
        </dependency>
    </dependencies>
</plugin>

You only need to adjust the next properties. We explain the most important of them. If you need deeper exploration of others You can find information about it on the http://jbehave.org/reference/stable/maven-goals.html.

    <!-- Default Jbehave plugin properties -->
    <jp.generate.view.after.stories>true</jp.generate.view.after.stories>
    <jp.ignore.failures>true</jp.ignore.failures>
    <jp.story.timeout.in.secs>800</jp.story.timeout.in.secs>
    <jp.threads>2</jp.threads>
    <jp.skip>false</jp.skip>
    <jp.stories>**/JbehaveLauncher.java</jp.stories>

The Most important:

- <jp.threads>

You can start tests in multythreading mode just by noticing count of threads in this parameter. Each thread was created for one story and after the story is finished the thread dies and at once the new thread will start and the new story will be performed. Each thread launches its own separate browser. In other words if you put 8 it means that at the same time will be launched 8 browsers and 8 story will be performed. Certainly if you have only for example 4 stories and put 8 only 4 threads will start.

- <jp.stories>

It is name of your class which provide you with our framework. You just need to extended our StoriesConfiguration.class by this class. See Runner class. You need to point out to name in the same format like this **/JbehaveLauncher.java.

- <jp.story.timeout.in.secs>

Is max time for running story. It is needed to avoid freezing of tests. If story lasts too much time It will be stopped after this time is over. To be honest it is just like an emergency exit. Such case is exceptional situation. Mostly If something goes wrong the framework will provide you with clear exceptions.

7.2.2. Processing of resources

For injecting maven parameters inside property files you need to add this in your pom.xml. Passing some properties over pom is a far more convenient then over property files. However by containing all properties in separated files we can control and interact with them much better due to the one entry point for it. See Property files.

<resources>
    <resource>
        <directory>${basedir}/src/main/resources</directory>
        <filtering>false</filtering>
        <excludes>
            <exclude>*.properties</exclude>
        </excludes>
    </resource>
    <resource>
        <directory>${basedir}/src/main/resources</directory>
        <filtering>true</filtering>
        <includes>
            <include>*.properties</include>
        </includes>
    </resource>
    <resource>
        <directory>${basedir}/src/main/stories</directory>
        <filtering>false</filtering>
    </resource>
</resources>

8. Properties

This framework provide you with possibility to use your property values in any places in your code. In order that it works you need to create property files with special names.

8.1. Property files

Lets have a look at location and names of properties files.

As you see there are 2 files which are located into resources. First file is for your own properties. Its name should be exactly configuration_custom.properties. Lets look into it and figure out how it works.

8.1.1. Custom properties

Take the property

time.wait.common = 30

How to get it in your code? Just get special object responsible for working with properties by writing

PropertyHolder propertyHolder = SpringUtil.getBean("propertyHolder");

Then you could get any values you have listed in the property file by writing for example

int COMMON_WAIT = Integer.parseInt(propertyHolder.getValue("time.wait.common"));

Other interesting case is properties are received from maven. Sometimes it is more convenient to perform it in this way, moreover there are situations where properties are different for launches (for example name of browser - 1 launch "Firefox", 2 launch "Chrome"). To adjust it you need to write in your property file something like this :

google.first.page = ${google.first.page}

Where ${google.first.page} is name property from your pom.xml file :

<properties>
    ...
    <!-- Default Running stories -->
    <google.first.page>https://www.google.by/</google.first.page>
    ...
</properties>

Don’t forget process your resources files as it is described into Processing of resources

Second file includes mandatory properties for correct work of framework. If you fill it out incorrectly You will get exceptions. For this reason be careful and pay attention to it. We try to explain to you that as clear as possible. Lets look into it.

9.1. Stories files

Lets have a look at a story and try to figure out it in more detail.

You could find huge amount of information about what is story is and which implementation of it is the best. It is a topic for doctoral dissertations. We won’t argue or prove something. Our approach has been quite good for us and was checked out with real project conditions. For all these reasons we suggest using this way. But each project is unique and it is very likely that you might prefer another way. It doesn’t matter. Feel free in this, the framework doesn’t have to restrict you in your willingness.

So, the story is like a Test Suite. Test Suite contains a lot of tests - Test Cases. Test Case in the BDD context - is a Scenario. Take it all into consideration the steps are just some actions which are performed by you only for one target - perform test case. They could have a big size or be separated to small actions, their count could be high or just a few steps, they could check something out or perform an action for moving ahead. It depends on the specific situation. Pay more your attention to the Scenario - it has to be logical and self sufficient Test Case. If a scenario fails you can just say that we was doing this (according to steps) and expecting the next (according to further steps). Each scenario will be launch even if scenario before it fails. And this is in accordance with logic about independence of Test Cases from each other.

10.2. Mapping Steps onto the Java code

It is very important part. Here is shown how to map your text steps onto java methods. Lets take a look at the next picture.

As We said before here Spring Framework, We are using here the Spring Dependency injection implementation. I try to lay out it simply without deep diving in this. It is like a container with a lot of java objects are created from your classes. These objects are SINGLETONS.

Note

We hope you know what singleton is and which tricks could appear during using them. For example using singletons in multythreading - It is one object and if you change something in it other thread will get it with your changes.

In order to put your object into there you have to write the next annotation @Component :

...
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {
....
}

If you would like to pull it or other objects out and start using them in your code please write another annotation @Autowired :

...
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }
    ....
}

Why do you need to put your Java Steps Class into container ? Because of Jbehave internal mechanism of mapping text steps onto java methods. Simply put Jbehave parse your step for example "When open google page" :

...
Scenario: google search page
When open google page
...

Then it goes to the "Container" and run through all methods with @When @Then annotations of the located there singletons trying to find necessary methods. Here it is @When("open google page").

...
import org.jbehave.core.annotations.Then;
import org.jbehave.core.annotations.When;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }

    @When("open google page")
    public void openGooglePage() {
        getPage().open();
        getPage().waitPageLoaded();
    }

    ....
}
Note

Therefore If inside container there is 2 or more the same methods it takes the first found one. If there is no appropriate method, you will be provided with exceptions.

10.3. Java page classes or Page Object pattern

10.3.1. Page Objects

As We said before here Selenium WebDriver, we are using WebDriver for our UI testing. There is a very famous and convenient pattern for working in close conjunction with it. It is called Page Object. It is very easy to figure it out. Your class just represents some Web Page, in other words Web Page is described by java code. Just take a look at the next picture and it gets a lot clearer.

As you see from the picture all you need to turn the Java class into the Page object is to extend our AbstractPage.class :

...
import com.epam.e3s.test.ui.framework.pages.AbstractPage;
import org.jbehave.web.selenium.WebDriverProvider;

public class GoogleFirstPage extends AbstractPage{

    public GoogleFirstPage(WebDriverProvider driverProvider) {
        super(driverProvider);
    }
    ...
}
...

Then you can get this page in any place of code by writing the next :

...
import com.epam.e3s.test.ui.framework.pages.Pages;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class GoogleFirstPageSteps {

    @Autowired
    private Pages pages;

    private GoogleFirstPage getPage() {
        return pages.getPage(GoogleFirstPage.class);
    }
    ....
}

Class Pages is a container for stocking all pages. When you creates your page first time it process your class, create Page Object, and put result into container. In further you will receive this page from container without extra workaround. By extending of AbstractPage.class you also get a lot useful methods for your work.

If you pay your attention to the names of methods, you are very likely to understand what they perform. Below We explore a few from them :

  • Open Page

...
public class GoogleFirstPage extends AbstractPage {
    ...

    public GoogleFirstPage open() {
        String urlBase = propertyHolder.getValue(PropertyConstants.GOOGLE_FIRST_PAGE);
        get(urlBase);
        return this;
    }
    ...
}

As you understand this method open page in browser. It performs the next method is taken from ancestor :

...
        get(urlBase);
...

  • Wait Page Load

...
    public static final String SEARCH_BAR_XPATH = "//input[@id='lst-ib']";

    public void waitPageLoaded() {
            waitVisibilityOfElementLocated(By.xpath(SEARCH_BAR_XPATH),
                                           PropertyConstants.COMMON_WAIT);
    }
...

Often it is very convenient to wait for Page will be loaded completely and only after it happens to try performing some actions on this page, such as getting some elements, pushing buttons and so on. And you as a developer of tests are much better aware what exactly "Page has been loaded" means for specific page. For example some bottom page elements appear or all javascript code has been performed and so on. In example we wait emergency of search bar on the page.

Here you also can see another method is taken from ancestor :

...
    waitVisibilityOfElementLocated(By.xpath(SEARCH_BAR_XPATH), PropertyConstants.COMMON_WAIT);
...

It is quite clear that this method waits some time until element gets visible on the page. If element doesn’t become visible during this time, you will be provided with an exception. Lets move on to the next methods. You can see the emergency of new interesting returned by methods object - WebElement.

  • Getting Web Elements

....
import org.openqa.selenium.By;
import org.openqa.selenium.WebElement;

...
    public WebElement getSearchBar() {
        return getVisibleElementLocated(By.xpath(SEARCH_BAR_XPATH),
                                        PropertyConstants.COMMON_WAIT);
    }

    public WebElement getSearchButton() {
        return getVisibleElementLocated(By.xpath(SEARCH_BUTTON_XPATH),
                                        PropertyConstants.COMMON_WAIT);

10.3.2. Web Elements

What is exactly Web page? Basically It is suite of high number of Web Elements. Such as buttons, inputs, dialog windows, paginators, bookmarks and so on, we can continue endlessly. Selenium WebDriver provides another abstraction which is called WebElement.class. It has some basic methods for working with it. You can browse these methods on your own. We just provide a little example of interaction with this element. So lets go on with code is taken from Mapping Steps onto the Java code. Put your attention to 2 last methods:

...

public class GoogleFirstPageSteps {

   @Then("write text '$text' into search bar")

As you see we are getting WebElements from page and try to interact with them. Everything is very easy. In other words you say to page : "The page give me the webElement search bar" and then you interact with it by calling sendKeys() - typing text inside search bar. Then you say to page : "Now The page give me the webElement search button" and push this button by calling click().

Now lets have a look at these methods which return our WebElements :

...
import org.openqa.selenium.By;
import org.openqa.selenium.WebElement;

public class GoogleFirstPage extends AbstractPage{

    public GoogleFirstPage(WebDriverProvider driverProvider) {
        super(driverProvider);

                                        PropertyConstants.COMMON_WAIT);

    public WebElement getSearchButton() {

In getSearchBar() we return WebElement Search Bar. Actually we do not know what is the element, it is just abstraction. We just wait for getting visible this element during some time and return this object without any clue what it can do or not - It supposes that who asked for this element is aware what this element do. In example above you saw that we typed text inside this element.

Also you can see such phrase as By.xpath(SEARCH_BAR_XPATH). What is Xpath? Please see very good and understandable article : http://www.w3schools.com/xsl/xpath_nodes.asp. Simply put It is address a wanted WebElement on the page. Knowledge about structuring it correctly is very important - but if you are QA and test Web part of application You must be familiar with it.

13. Passing VARIABLES through steps into java methods

From our point of view in Jbehave there is one big problem. It doesn’t provide possibility to save and contain reusable variables. In order to resolve it we developed our own mode to create and reuse them in any place in the code. Imagine it like you are putting your variables in one big Map which is available in code. As result you could gain variables by key in any place. Also bear in mind that if you change them in some step in the next step you will get changing version of it. This container is ContexVariableTestData.class and you get it in any your component by injecting it as it is shown below. It has a few very convenient methods such as getContextVariable() (returns object and you can make a type conversion into your object).

Java code inside java steps classes :
...
import com.epam.e3s.test.ui.framework.aspect.provider.ContextVariableTestData;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;

@Component
public class SomeClassSteps {

    @Autowired
    private ContextVariableTestData context;

    @Given("set some object")
    public void saveContextVariable() {
        MyObject objsect = new MyObject()
        String name = "Variable name";
        context.setContextVariable(name, value);
    }

    @Given("get some object")
    public void saveContextVariable() {
        String name = "Variable name";
        MyObject objsect = (MyObject) context.getContextVariable(name);
    }

}

14. WebElements Approach

As you remember from Java page classes or Page Object pattern there is a special structure of layer which is responsible for interacting with WebDriver. We have described 2 interesting objects there - Page and WebElement object. Also We have given a few simple examples. Here Web Elements we said that there are some WebElements out of box with some default methods : click(), sendText() and etc. But the modern WebPage is a far more complex and can consist of huge count of elements. And if we return from page one WebElement per one method it will get very inconvenient for supporting and expanding our tests. Following this We wold like to propose the next approach.

Why don’t you create you own WebElements? Which is like a page are able to return other elements or perform some specific actions are relevant to it.For understanding it Lets take a look at java code examples :

...
@Component
public class GoogleSecondPageSteps {
    ...

    @Then("the title of '$number' result on the Google search page is '$result'")
    public void checkingTitleFirstResult(String number, String result) {
        Assert.assertEquals(getPage().getResultTitle(number), result);
    }

    // WebElements Approach
    @Then("the title of '$number' result on the Google search page is '$result' using WebElements approach")
    public void checkingTitleFirstResultWebElementApproach(String number, String result) {
        Assert.assertEquals(getPage().getResultTitleWebElementApproach(number), result);
    ...
}

In these two methods we are doing the same things but in different ways. We are getting the Title of search result form google page.

First method is corresponding to the next java code :

...
public class GoogleSecondPage extends AbstractPage{

    public static final String SEARCH_RESULTS_XPATH = "//div[@id='ires']";
    public static final String RESULT_XPATH_FROM_PAGE = SEARCH_RESULTS_XPATH + "//div[@class='g'][{0}]//a";

    public GoogleSecondPage(WebDriverProvider driverProvider) {
        super(driverProvider);
    }

    public String getResultTitle(String numberOfResult) {
        By by = formatXpath(RESULT_XPATH_FROM_PAGE, numberOfResult);
        return getVisibleElementLocated(by, PropertyConstants.COMMON_WAIT).getText();
    }
...

It is a usual way. We are getting visible WebElement (Title) and extracting text from it. Everything is fine and correct. But lets imagine you need to check text, data, and another one and another one on this search result. It is very likely that your page object will contain about 10 methods only for checking 1 WebElement. But there are a lot other ones. How do you feel about java page object with 50 - 80 methods? Is it fine with you? Certainly it’s not. In addition what if this WebElement are used in another place, for example on the third page. You need to write these methods again. Not a good idea.

For all these reasons the best way is to separate responsibilities by creating one own WebElement : Search Result.

package com.epam.e3s.test.ui.pages.google.secondpage.webelements;

import com.epam.e3s.test.ui.PropertyConstants;
import com.epam.e3s.test.ui.framework.webelement.WebElementWrapper;
import org.openqa.selenium.By;

public class ResultOnThePage extends WebElementWrapper {

    public static final String TITLE_XPATH = ".//h3/a";
    public static final String HIGHLIGHTED_XPATH = ".//em";
    public static final String SHORT_DESCRIPTION =  ".//span[@class='st']";

    public ResultOnThePage(WebElementWrapper webElementWrapper) {
        super(webElementWrapper);
    }

    public String getTitle() {
        return getVisibleElementLocated(By.xpath(TITLE_XPATH),
                                        PropertyConstants.COMMON_WAIT).getText();
    }

    public String getHighlightedText() {
        return getVisibleElementLocated(By.xpath(HIGHLIGHTED_XPATH),
                                        PropertyConstants.COMMON_WAIT).getText();
    }

    public String getShortDescription() {
        return getVisibleElementLocated(By.xpath(SHORT_DESCRIPTION),
                                        PropertyConstants.COMMON_WAIT).getText();
    }
}

For it you need to extend our WebElementWrapper.class. Then you can work like with page object. Here We create 3 methods and hope that the main principle get clear to you.

Note

One important thing to note is the writing xpath. By putting in the beginning "." (dot) you make your WebElement like a box. Other elements were looked for only inside this WebElement.

What is Grid please see http://selenium2.ru/docs/selenium-grid.html.

As we mentioned here WebDriver properties you can use Grid implementation for launching your tests.

source,xml,linenums]

- <config.url.gridhub>http://localhost:4444/wd/hub</config.url.gridhub>

If it is true you are going to perform tests using Grid implementation.

In the folder Grid you find different files for performing it. We created these .bat files special for windows system. From our point of view to launch browser in the Windows is closer to real using. In a word there are a huge count of ways to launch GRID. You can launch only one hub and many different nodes or you can launch many hubs and many nodes. It all depends on your project demands and ability of infrastructure. We went for 2 different hubs which connected only with one type of browser (Chrome, Firefox).

So for successful performing it you need :

  • transfer the folder grid into your Windows server

  • run run_CHROME_HUB.bat file (for chrome), run_FIREFOX_HUB.bat file (for firefox). It launches one hub and 1 node with 10 instances on your server and links them between each other. Don’t worry about instances - it doesn’t mean that you start 10 browsers. You will start them on demand. If test get started and needs a browser in this way the browser will be launched but no more then 10. You can it customize on your owm - it is easy.

  • allow to connect to this server other machine (turn off firewall for special machines)

Note

Certainly you need installed browser at the server before running these scripts.

That is it. Then just set the address of this server into the pom file in <config.url.gridhub> property - like it was showed above.

We would like to notice that when you are using GRID we need to use other implementation of BrowserMob Proxy due to incompatibility of old one. Actions described above perform it and you don’t spot any changes for your work. This process will work like a daemon.
